# 中文写作技能指南

这份指南旨在指导大模型生成高质量、现代风格的简体中文内容。适用于日报、博客文章、科技资讯及评论类文章等场景。

核心原则是：**简单 (Simple)**、**人性化 (Human)**、**清晰 (Clear)**。

## 1. 核心基调与原则 (Voice & Tone)

写作不仅仅是传递信息，更是与读者建立联系。我们的声音应当是：

- **温暖轻松 (Warm and relaxed)**：像人一样说话，自然、友好，而不是冷冰冰的机器或高高在上的官僚。要有同理心。
- **清晰明了 (Crisp and clear)**：直击要点，避免废话。便于读者快速扫描和理解。
- **乐于助人 (Ready to lend a hand)**：以读者为中心，预判他们的需求，提供切实可行的帮助。

## 2. 通用写作建议 (General Writing Tips)

- **像说话一样写作**：使用日常对话中的词汇。写完后大声朗读，如果听起来像是在演讲或读公文，那就改得更口语化一些。
- **减少套话和学术腔**：
  - 避免"很有代表性"、"值得注意"、"不可忽视"等空洞评价，直接说具体特点。
  - 慎用"本质上是"、"其核心在于"，改为更直接的表述（如"核心是"、"关键是"）。
- **保持简洁**：
  - **一句话只讲一个意思**。
  - 删掉无意义的修饰词（如"非常"、"极其"、"显然"）。
  - 避免冗长的从句和复杂的句式。
- **直奔主题**：采用"倒金字塔"结构，把最重要的信息（结论、亮点）放在最前面。
- **以读者为中心**：
  - 多用**第二人称"你"**，少用第一人称"我们"或"我"。
  - 关注读者能获得什么价值，而不是单纯描述事物本身。
  - **示例**："这款工具能帮你快速生成文章摘要" (好) vs "这款工具具有文章摘要生成功能" (差)。
- **使用主动语态**：让主语执行动作。中文应尽量避免滥用"被"字句。
  - **示例**："OpenAI 发布了新模型" (好) vs "新模型被 OpenAI 发布" (差)。
- **正面表述**：尽量告诉读者"是什么"、"有什么价值"，而不是"不是什么"。
- **避免绝对化表述**：技术讨论中慎用"都是"、"从来不是"、"真正"等绝对化词汇。可用"在复杂系统领域"、"多数情况下"等限定语增加表述的准确性。

## 3. 内容结构与排版 (Structure & Formatting)

良好的排版能帮助读者快速获取信息。

- **标题 (Headings)**：
  - 使用有意义的标题，概括段落内容。
  - 标题应简短有力，尽量不使用标点符号（问号除外）。
  - 使用层级结构（H1, H2, H3）来组织内容。
- **列表 (Lists)**：
  - **无序列表**：用于列举同类项目。
  - **有序列表**：仅用于表示步骤、顺序或优先级。
  - **平行结构**：列表项的语法结构要保持一致（例如，都以动词开头或都是名词短语）。
- **段落 (Paragraphs)**：
  - 保持简短，每段建议不超过 4-6 行。
  - 一个段落专注于一个主题。
- **强调**：
  - 关键术语、重要概念建议使用 **加粗**。
  - 避免使用全大写（针对英文缩写）或过多的感叹号来表示强调。

## 4. 词汇与语法规范 (Grammar & Usage)

- **中英文混排 (Pangu Spacing)**：
  - 英文单词/数字与中文字符之间**必须保留一个空格**，以提高可读性。
  - **示例**：在 Windows 中设置... / 上传 5 张图片...
  - **例外**：百分号、度数符号等单位符号与数字之间不需要空格（如 50%）。
- **避免行话 (Jargon)**：
  - 除非读者是特定领域的专家，否则尽量避免使用晦涩的技术术语。
  - 如果必须使用，请在第一次出现时用通俗易懂的语言解释。
- **标点符号**：
  - 使用全角标点符号（逗号，、句号。、冒号：等）。
  - **例外**：在行内代码、数学公式或特定技术引用中，使用半角符号。
  - 英文缩写词后通常不需要加句号（如 "App" 而不是 "App."）。

## 5. 日报写作专项 (Daily Newsletter Writing)

### 5.1 标题创作

- 简洁有力，15 字以内为佳。
- 突出核心信息或亮点，避免堆砌关键词。
- 可用数字增强吸引力（如"本周 5 个值得关注的 AI 工具"）。
- **禁止标题党**：不夸大、不误导。

### 5.2 开头与结尾

- **开头**：
  - 使用一级标题 `# 本期导读`。
  - 随后编写一段简介，概述当天的主要技术趋势或热门话题，引导读者快速定位感兴趣的内容。
- **结尾**：
  - 使用一级标题 `# 尾巴`。
  - 编写简短的总结陈词，回顾当天的核心价值，并固定结尾语（如“我们下期再见”）。

### 5.3 分类板块

- 使用二级标题 ## 进行分类，有两种方式可选：
  - **方式一（来源导向）**：
    - ## 资讯：侧重于新闻、发布声明、工具更新等。
    - ## 博客：侧重于深度文章、技术教程、观点评论等。
  - **方式二（主题导向）**：按技术主题聚类，如"基础设施可靠性"、"隐私与安全"、"AI 与工具"等。
- **选择建议**：当文章间关联性强时优先用主题导向，帮助读者建立知识连接；当内容较分散时用来源导向。

### 5.4 条目撰写（资讯/博客）

- 每个条目使用三级标题 ###
- 每个条目结构：**标题 + 3-6 句摘要**（链接嵌入标题或关键词）。
- **摘要要灵活多变**，根据内容特点选择合适的切入角度，避免千篇一律：
  - 新产品/工具：是什么 → 核心功能 → 使用场景
  - 模型发布：性能亮点 → 对比提升 → 适用场景
  - 行业动态：事件本身 → 影响/意义 → 后续展望
  - 技术突破：解决什么问题 → 技术原理简述 → 实际应用
  - 融资/收购：金额/估值 → 公司背景 → 行业信号
  - 开源项目：项目定位 → 特色功能 → 上手难度
- **写法示例**（同一条新闻的不同风格）：
  - 功能导向：`[Cursor](url) 新增 Agent 模式，可自动执行多步骤编程任务。`
  - 场景导向：`写代码时想偷懒？[Cursor](url) 的 Agent 模式能帮你一键搞定重构。`
  - 对比导向：`[Cursor](url) 更新后，Agent 能力已接近 Claude Code。`
- 避免复制原文，用自己的话提炼核心信息。
- **链接嵌入规范**：
  - 原文链接必须保留，但**不要单独列出**。
  - 将链接作为**锚文本**嵌入段落，锚文本应贴近标题或核心名词。
  - 锚文本优先使用：产品名、项目名、公司名、论文标题等。
  - 使用正式名称（首字母大写），避免使用 URL slug 或过小写的标识符。
  - ✅ 好的示例：`[Palantir Ontology Strategy](https://github.com/...)`
  - ❌ 差的示例：`[palantir-ontology-strategy](https://github.com/...)`
  - 格式统一：链接与周围文字之间**不留空格**，如 `[标题](url)` 而非 `[ 标题 ]( url )`。
  - ✅ 好的示例：`[Sora](https://openai.com/sora) 是 OpenAI 推出的视频生成模型，支持长达 60 秒的高清视频生成。`
  - ❌ 差的示例：`Sora 是 OpenAI 推出的视频生成模型。[原文链接]`
  - ❌ 差的示例：`Sora 是 OpenAI 推出的视频生成模型。阅读更多 →`
  - ❌ 差的示例：`[ Sora ]( https://openai.com/sora )`（多余空格）

### 5.5 HN 评论反响处理

日报的核心价值之一是节省读者翻阅评论的时间，需专门提炼 HN 社区的观点：
- **提炼共识**：总结高赞评论的普遍观点（如"核心批评是..."、"大家并不反对 X，而是关注 Y"）。
- **呈现分歧**：如有明显对立观点，简要呈现不同立场及其论据。
- **技术深度**：HN 评论常包含工程师的实际经验，优先保留有技术细节的观点。
- **避免堆砌**：不必罗列所有观点，选择最有洞察力的 1-2 个角度即可。
- **明确标注来源**：
  - 必须区分**原文观点**、**作者解读**和**HN 社区观点**。
  - HN 观点用明确标识："HN 讨论认为..."、"评论区的共识是..."、"HN 提醒读者..."
  - 避免模糊的"这提醒我们"，如果是指 HN 观点，应写"HN 讨论提醒读者..."
  - 个人解读可用"简评"、"值得注意的是"等前缀，但与 HN 观点必须区分

### 5.6 信息整合与引用

- 多源信息需整合归纳，避免简单罗列。
- 引用时注明来源（网站名或作者）。
- 区分事实陈述与个人观点。

### 5.7 摘要提炼（Summary）

- 用 1-2 句话概括全文精华。
- 突出本期最大亮点或主题。
- 适合用于 RSS、社交媒体分享。

### 5.8 格式一致性检查清单

生成日报后，对照以下清单检查格式统一性：

**Frontmatter 格式**
- [ ] `title` 值必须使用双引号包裹：`"Hacker News 日报 (YYYY-MM-DD)"`
- [ ] `tags` 格式统一（列表形式或行内形式，整站保持一致）

**结构统一**
- [ ] 导读后直接接分类，**不要**加"文章梗概和评论反响"这类冗余标题
- [ ] 分类标题统一：`## 资讯`、`## 博客`（或按主题聚类）
- [ ] 可选板块：如当期有明显跨文章趋势，可在尾巴前加 `## 今日观察`

**链接格式**
- [ ] 锚文本与括号之间**不留空格**：`[标题](url)` ✓ `[ 标题 ]( url )` ✗
- [ ] 使用正式产品/项目名称作为锚文本，避免 URL slug

**HN 评论处理统一**
- [ ] 明确区分原文观点、作者解读、HN 观点
- [ ] 使用统一标识语："HN 讨论认为..."、"评论区共识是..."
- [ ] 如需放 HN 链接，统一放在条目末尾：`讨论见 [Hacker News 帖子](url)`
- [ ] 优先提炼具体观点，而非仅放链接

**条目结构（可选但建议统一）**
- 技术/架构类文章可采用：
  - 核心内容 → 要点列表 → 简评（含 HN 观点）
- 观点/讨论类文章可采用：
  - 核心内容 → 简评（直接融入 HN 观点）

## 6. 文章写作技巧 (Article Writing Tips)

### 6.1 叙事节奏与可读性

- 段落简短（4-6 行），避免大段文字堆砌。
- 使用小标题打破长文，引导阅读节奏。
- 适当使用过渡句，保持行文流畅。

### 6.2 观点表达与论证

- 观点明确，不要模棱两可。
- 有理有据：用事实、数据、案例支撑观点。
- 保持客观中立，避免情绪化表达。

### 6.3 过渡与衔接

- 条目之间可用简短过渡语串联（如"除了模型更新，本周工具类也有不少亮点"）。
- 避免生硬跳转，保持阅读连贯性。
- **分类之间可加过渡段**：在"资讯"、"博客"等分类前用 1-2 句话说明本组内容的共同主题，帮助读者建立预期。
- **考虑删除冗余标题**：如"文章梗概和评论反响"这类 merely descriptive 的标题可直接省略，让内容更紧凑。
  - 判断标准：如果标题只是重复结构信息（如"以下是文章列表"），而不能概括内容主题，则应删除。
  - 删除后直接用二级分类标题（如"## 资讯"、"## 博客"）承接"# 本期导读"，节奏更紧凑。

## 7. 改写示例 (Examples)

以下是一些优化前后的对比，帮助你更好地理解上述原则：

| 场景     | 优化前 (Bad)                                          | 优化后 (Good)                                           | 原因                         |
| :------- | :---------------------------------------------------- | :------------------------------------------------------ | :--------------------------- |
| **标题** | OpenAI 发布了一个非常强大的全新 AI 模型               | OpenAI 发布 GPT-5，推理能力大幅提升                     | 去除冗余修饰，突出核心信息。 |
| **摘要** | 这篇文章介绍了一个工具，它可以帮助你做很多事情        | 一款开源的 AI 图像生成工具，支持本地部署                | 直接说明是什么、有何特点。   |
| **开头** | 大家好，欢迎来到新一期的日报，今天我们将为大家带来... | 昨天 AI 圈最大的新闻：Claude 4 正式发布。               | 直奔主题，抓住读者注意力。   |
| **引用** | 根据网上的消息，这个模型很厉害                        | 据 Anthropic 官方博客，Claude 4 在推理任务上超越 GPT-4o | 注明来源，表述具体。         |
| **排版** | 注意：这个工具非常非常重要！！！                      | **亮点**：这款工具值得关注。                            | 避免过多感叹号，用加粗强调。 |
